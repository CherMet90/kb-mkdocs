# OpenConnect

`ocserv` — сервер OpenConnect VPN для удалённого доступа к сети. Обычно используется, когда нужно:

- выдавать доступ по пользователям и паролям;
- разделять пользователей по группам;
- назначать разные маршруты и DNS для разных групп;
- поддерживать split tunneling;
- централизованно управлять доступом к внутренним ресурсам.

## Основные файлы

Чаще всего используются:

- `/etc/ocserv/ocserv.conf` — основной конфиг сервера;
- `/etc/ocserv/ocpasswd` — база пользователей при аутентификации через `ocpasswd`;
- `/etc/ocserv/config-per-group/` — отдельные настройки для групп.

Пример структуры:

```bash
/etc/ocserv/config-per-group/
├── partners
└── staff
```

## Что такое группа в ocserv

Группа в `ocserv` — это логическая группа доступа на стороне VPN-сервера.

Она используется для применения отдельных параметров:

- маршрутов;
- DNS;
- split tunneling;
- ограничений доступа;
- других настроек подключения.

## Создание новой группы

Если в конфигурации используется `config-per-group`, то для новой группы достаточно создать файл с её именем:

```bash
sudo touch /etc/ocserv/config-per-group/newgroup
sudo chmod 600 /etc/ocserv/config-per-group/newgroup
sudo chown root:root /etc/ocserv/config-per-group/newgroup
```

Имя файла должно совпадать с именем группы, которую сервер видит при аутентификации пользователя.

## Пример настроек группы

Ниже пример безопасно обезличенной конфигурации для публичного репозитория:

```ini
ipv4-network = 10.250.10.0
ipv4-netmask = 255.255.255.0

tunnel-all-dns = false
dns = 10.250.53.10
dns = 10.250.53.11
split-dns = corp.example
split-dns = vpn.example.internal

# Только эти сети видны через VPN
route = 10.250.53.0/24
route = 10.250.100.0/24

# Запретить клиенту ходить куда-либо ещё через туннель
restrict-user-to-routes = true
```

Что здесь важно:

- `ipv4-network` и `ipv4-netmask` задают пул адресов для клиентов;
- `dns` определяет DNS-серверы, которые получает клиент;
- `tunnel-all-dns = false` оставляет DNS-трафик вне полного туннеля, если это требуется вашей схеме;
- `split-dns` отправляет запросы только для указанных зон через VPN-DNS.

## Добавление пользователя в группу

Если используется `ocpasswd`, пользователь создаётся так:

```bash
sudo ocpasswd -c /etc/ocserv/ocpasswd -g newgroup username
```

Если пользователь уже существует, можно изменить его принадлежность к группе той же командой с другим значением `-g`.

## Применение изменений: reload и restart

В большинстве рабочих сценариев я использую:

```bash
sudo occtl reload
```

Это удобно, когда вы меняете:

- `config-per-group`;
- список пользователей;
- часть параметров авторизации и политики доступа.

### Ограничения `occtl reload`

`reload` не стоит считать универсальной заменой перезапуска.

Обычно он полезен для мягкой перезагрузки конфигурации без полной остановки сервиса, но:

- не все параметры применяются к уже активным сессиям;
- изменения сетевых параметров, портов, IP-пула, TLS-сертификатов или интеграции с firewall могут требовать `restart`;
- если нужно гарантированно применить изменения ко всем новым подключениям и избежать расхождения состояния, лучше выполнить полный перезапуск и переподключение клиентов.

Практическое правило:

- **`occtl reload`** — для точечных изменений конфигурации;
- **`systemctl restart ocserv`** — когда меняется поведение сервиса, сеть или есть сомнения, что reload достаточно.

## Система банов

В `ocserv` реализована IP-based система банов с накопительным счётом (score).
Каждый новый TCP-коннект или неудачная аутентификация добавляет IP-адресу
определённое количество очков. При превышении порога `max-ban-score` IP
попадает в бан-лист, и **новые** подключения с него блокируются до истечения
времени бана.

**Ключевая особенность (v1.2.4):** бан не разрывает уже активные сессии —
блокируются только попытки установить новые соединения. Поэтому один и тот же
IP может одновременно присутствовать и в списке забаненных (`occtl show ip bans`),
и среди активных пользователей (`occtl show users`).

### Параметры банов (v1.2.4)

В версии 1.2.4 нет отдельного параметра `ban-time` — длительность бана задаётся
через `min-reauth-time`.

| Параметр | Назначение |
|---|---|
| `max-ban-score` | Порог срабатывания бана (суммарный счёт) |
| `ban-reset-time` | Окно, через которое накопленные очки обнуляются (если не достигнут порог) |
| `ban-points-connection` | Очки за каждое новое TCP-подключение (начисляются до аутентификации) |
| `ban-points-wrong-password` | Очки за неверный пароль (дополнительно к очкам за подключение) |
| `min-reauth-time` | Минимальное время до повторной попытки аутентификации **и** длительность бана |

Таким образом, схема начисления для одного соединения с неверным паролем:

```
ban-points-connection + ban-points-wrong-password
```

### Сброс счётчика

Счётчик очков по IP обнуляется, если за время `ban-reset-time` не было
превышения `max-ban-score`. Если порог превышен — IP блокируется на
`min-reauth-time` секунд, и дальнейшие обращения только продлевают бан.

## Бан-политика ocserv

Параметры подобраны так, чтобы 10 реконнектов за 20 минут от одного IP
гарантированно не приводили к бану, но чтобы быстрый перебор паролей или
шторм подключений пресекались.

```ini
# /etc/ocserv/ocserv.conf
max-ban-score = 100
ban-reset-time = 1200
ban-points-connection = 10
ban-points-wrong-password = 20
min-reauth-time = 10800   # в v1.2.4 это же длительность бана (3 часа)
auth-timeout = 60
```

| Сценарий | Результат |
|---|---|
| 10 реконнектов за 20 мин | 100 очков — **бан** (3 часа) |
| 8–9 реконнектов за 20 мин | 80–90 очков — норма (не бан) |
| 1 неверный пароль | 30 очков |
| 5 неверных паролей за 20 мин | 150 очков — **бан** |
| Медленный сканер: ≤5 TCP-стуков за 20 мин | ≤50 очков — не бан |

**Важно:** после истечения бана (3 часа) IP должен «заслужить» новый бан
с нуля. Если IP продолжает стучать, он снова накопит очки и уйдёт в бан.

## Управление банами

### Снять бан с одного IP

```bash
sudo occtl unban ip 203.0.113.5
```

Счётчик очков и бан сбрасываются. Активная сессия (если есть) не прерывается.

## Тонкая настройка таймаутов

### `auth-timeout`

Максимальное время (в секундах) на прохождение аутентификации после
установки TCP-соединения. Если клиент не успел — соединение разрывается.

Рекомендация: 60 секунд. Этого достаточно для ввода логина/пароля и OTP.

```ini
auth-timeout = 60
```

### `idle-timeout` и `mobile-idle-timeout`

Отключают клиента при отсутствии трафика. Не рекомендуются, если цель —
сократить число реконнектов: каждое отключение по таймауту порождает
новое переподключение и начисление `ban-points-connection`.

### `session-timeout`

Принудительный разрыв сессии независимо от активности. Добавляет
гарантированный реконнект каждому клиенту раз в N секунд. Использовать
только при жёстких политиках ротации сессий.

## Проверка настроек

После изменения конфигов проверьте сервис:

```bash
sudo systemctl status ocserv
```

Полезно также смотреть логи:

```bash
journalctl -u ocserv -f
```

## Полезные команды

Список пользователей:

```bash
sudo cat /etc/ocserv/ocpasswd
```

Посмотреть активные сессии:

```bash
sudo occtl show users
```

Посмотреть список забаненных IP:

```bash
sudo occtl show ip bans
```

Посмотреть IP с ненулевым счётом (ещё не забаненные):

```bash
sudo occtl show ip ban points
```

Снять бан с одного IP:

```bash
sudo occtl unban ip <IP-адрес>
```

Проверка процесса:

```bash
ps aux | grep ocserv
```

Мягкая перезагрузка конфигурации:

```bash
sudo occtl reload
```

Полный перезапуск сервиса:

```bash
sudo systemctl restart ocserv
```
