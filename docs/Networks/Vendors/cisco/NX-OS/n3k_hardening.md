# Харденинг Cisco NX-OS (N3K)

Документация описывает базовые меры по защите Management Plane на коммутаторах Cisco Nexus 3000

## 1. Системное логирование и ACL Log
Установка корректных уровней логирования для `acllog` и сервера Syslog позволяет детально отслеживать события безопасности и попытки несанкционированного доступа.

```cisco
logging level acllog 6
logging server <SYSLOG_SERVER_IP> 6 use-vrf default facility local6
```
- `acllog 6` (informational): включает логирование срабатываний `deny`-правил, которые явно содержат параметр `log`.
- `facility local6`: рекомендуется для выделения Cisco-устройств в отдельный поток, что упрощает фильтрацию и ротацию на стороне Syslog-сервера.

## 2. Ограничение доступа к VTY
Применение ACL к виртуальным терминалам строго регламентирует, какие IP-адреса могут подключаться к CLI коммутатора. Это первая линия обороны плоскости управления.

```cisco
ip access-list ACL-VTY-ACCESS
  remark === mgmt net ===
  permit ip any <подсеть vrf management>
  remark === global admin nets ===
  permit ip 10.10.1.0/24 any
  permit ip 10.10.2.0/24 any
  permit ip 10.10.3.0/24 any
  permit ip 10.10.4.164/31 any
  permit ip 10.10.4.248/30 any
  permit tcp 10.10.5.10/32 any eq 22
  remark === log drops ===
  900 deny ip any any log

line vty
  access-class ACL-VTY-ACCESS in
```
- Последовательность `900` гарантирует, что все несопоставленные соединения логируются перед сбросом.
- `access-class ... in` применяется исключительно к входящим сессиям (SSH/Telnet).

## 3. Защита SSH-сессий

```cisco
ssh login-gracetime 15
ssh idle-timeout 10
```
- `login-gracetime 15`: ограничение времени на ввод учетных данных
- `idle-timeout 10`: принудительный разрыв соединения после 10 минут бездействия. Освобождает пул VTY для легитимных администраторов.

## 4. Изоляция NXAPI и Web-интерфейса
По умолчанию HTTP/HTTPS и NXAPI могут быть доступны в глобальной VRF

```cisco
nxapi use-vrf management
```
- **Trade-off:** После применения доступ к веб-интерфейсу и API будет возможен только через интерфейсы, привязанные к `management` VRF

## 5. ACL для SNMP
SNMP является частым вектором атак при слабой защите. Рекомендуется ограничить доступ к агенту конкретным IP-адресом системы мониторинга.

```cisco
ip access-list 15
  permit udp 10.10.6.92/32 any eq snmp
```
- Данный ACL создает правило доступа. Для его активации он должен быть явно привязан к конфигурации SNMP-сообщества (`snmp-server community ... use-ipv4acl 15`)

## 6. CoPP (Control Plane Policing) для защиты CPU
Политики CoPP защищают процессор коммутатора от атак типа DoS и перебора паролей, ограничивая трафик, предназначенный контрольной плоскости.

```cisco
policy-map type control-plane copp-system-policy
  class copp-ssh
    police pps 15 
```
- `class copp-ssh`: класс, обрабатывающий входящие SSH пакеты.
- `police pps 15`: жесткий лимит в 15 пакетов в секунду. Легитимный CLI-доступ требует минимальной пропускной способности на уровне пакетов, поэтому ограничение эффективно отсекает сканеры и брутфорс, не влияя на работу администраторов.
- **Важно:** Убедитесь, что на устройстве активирован кастомный профиль CoPP перед применением изменений, чтобы не перезаписать или нарушить работу стандартных политик безопасности.
