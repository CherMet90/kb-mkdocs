# Zabbix

## Где что искать

### Активные проблемы
`Monitoring → Problems`

Тут смотри:

- какие алерты сейчас активны
- что уже подтверждено
- что подавлено
- что можно закрыть вручную


### Последние значения метрик
`Monitoring → Latest data`

Нужен для:

- поиска конкретного хоста
- просмотра всех items
- проверки, какие данные реально собираются
- поиска точного имени item перед настройкой дашборда

<!-- more -->
### Хосты и шаблоны
`Configuration → Hosts`  
`Configuration → Templates`

Здесь ищутся:

- привязанные шаблоны
- discovery rules
- триггеры
- макросы хоста
- item'ы и графики конкретного устройства


### Автообнаружение
`Configuration → Hosts → <host> → Discovery rules`  
или  
`Configuration → Templates → <template> → Discovery rules`

Здесь правятся:

- фильтры discovery
- trigger prototypes
- item prototypes
- macros, влияющие на сгенерированные сущности


### Уведомления и действия
`Alerts → Actions`

Тут важно проверять:

- какие группы участвуют в action
- какие условия завязаны на host group
- не сломает ли новая группа чужую схему уведомлений


### Права доступа
`Administration → User groups`

Проверять, если надо понять:

- кто видит новые группы
- кому доступны новые хосты
- не нужно ли расширить права на новую ветку групп

## Как читать проблему в Zabbix

Если прилетела непонятная проблема, сначала иди по такому маршруту:

1. `Monitoring → Problems`
2. Открой проблему
3. Посмотри имя хоста и имя триггера
4. Перейди в `Configuration → Hosts → <host>`
5. Открой `Triggers`
6. Найди исходный триггер или trigger prototype
7. Смотри expression, recovery expression и описание

Это самый быстрый способ понять, откуда взялся алерт.

## How-to

### Как посмотреть все метрики конкретного хоста

1. Зайди в `Monitoring → Latest data`
2. В фильтре выбери нужный `Host`
3. При необходимости уточни `Name`
4. Нажми `Apply`
5. Открой нужный item и проверь `Last check`, `Last value`, график и историю

### Как закрыть ложную проблему вручную

Подходит, если триггер разрешает ручное закрытие.

1. Зайди в `Monitoring → Problems`
2. Найди нужную проблему
3. Нажми `Acknowledge`
4. Добавь комментарий
5. Включи `Close problem`
6. Сохрани

### Как временно скрыть проблему

Если событие пока не чинится, но мешает шумом:

1. Зайди в `Monitoring → Problems`
2. Открой нужную проблему
3. Используй `Suppress`
4. Выбери период подавления
5. Сохрани

### Как понять, откуда взялся алерт по интерфейсу

1. Зайди в `Monitoring → Problems`
2. Открой проблему по интерфейсу
3. Скопируй имя триггера
4. Перейди в `Configuration → Hosts → <host> → Triggers`
5. Найди этот триггер
6. Посмотри expression и описание

### Как отключить `Link down` для конкретного Wi-Fi интерфейса

Это работает только в шаблонах, где триггер использует контекстный macro вида `{$IFCONTROL:"{#IFNAME}"}`.

1. Зайди в `Configuration → Hosts`
2. Открой нужный MikroTik
3. Перейди на вкладку `Macros`
4. Добавь macro with context
5. Укажи точное имя интерфейса или регулярку
6. Поставь значение `0`
7. Сохрани хост
8. Закрой старую проблему в `Monitoring → Problems`

Пример:

```text
{$IFCONTROL:regex:"^(5GHZ|2GHZ|wifi).*"}=0
```

### Как не спамить по портам, которые должны быть admin down

Если порт заведомо не используется, логика мониторинга должна быть такой:

1. Порт остается administratively down
2. Триггеры на `Link down` для него не нужны
3. Если алерт уже завис, его можно закрыть вручную
4. Если порт когда-то понадобится — его поднимают и начинают мониторить как рабочий
